# 老師後台（Teacher Cohort）PRD

> 狀態：**規劃中（planned）**——截至 2026-07-16，全系統無任何老師後台程式碼（見「現況盤點」）。
> 本文件由 2026-07-16 的規劃討論產出，回填 `prd/learning-ecosystem.md` 點名的結構性空白：「導師供給機制是整個引導層唯一沒有人負責思考的部分」。

## 核心概念

給課程講師的**陪跑教練台，不是 LMS**。島島的定位是學習連接器，核心有機質是主題實踐（practice）打卡。老師後台的價值不是上傳教材、出考卷、算成績，而是：

**讓老師看見學員課後的學習節律，並在對的時機回應。**

這直接承接 GTM「B 軌」策略（`docs/product/gtm/plg-gtm-strategy.md`）：講師的痛點是完課率與續購，島島賣的是「學員陪跑空間」——講師把課程作業做成實踐模板 → 學員複製打卡 → 講師看全班節律 → 期末批次匯出成果當招生素材。

## 使用者問題

**講師端**（付費者）：
- 課程結束後學員練習斷裂，完課率低、續購難
- 看不見學員課後有沒有真的在做，只能在群組裡乾喊
- 招生缺乏「學員真實成果」的素材

**學員端**（被帶進來的使用者）：
- 課後練習孤立無援，沒有節律、沒有回饋
- 注意：這群人是**被指派的使用者**（動機外部），與島島現有自願用戶行為模式不同——設計必須考慮轉化橋（見「北極星指標」）

## 設計原則（不可退讓）

1. **量流動、不量壓力**（承島島指標戒律）：儀表板呈現流動、引導回應；**永不**呈現缺勤排名、連續天數、學習時長、deadline 達成率。「需要關心名單」只呈現流動中斷＋該學員最後一則打卡，行動按鈕是「送個鼓勵」不是「發提醒」。
2. **活動內容預設不進公共靈感牆**：只進活動牆，學員可自選公開——保護原生社群純度，也是隱私邊界。
3. **老師看到的是「活動內的產出」，不是「學員這個人」**（見下節可見性模型）。

## 資料可見性模型

可見性的錨點是**練習與活動的綁定關係**，不是使用者。推導鏈：

```
老師 ──(cohort_members.role = owner)──▶ 活動(cohort)
活動 ──(cohort_practices 綁定)──────▶ 學員的 practice
practice ──(隸屬關係)──────────────▶ 打卡(check-ins)
```

**可見性 = 「owner 身分」×「活動綁定」×「關係存續」三者同時成立**，任一失效即不可見。

### 資料分界

| 老師看得到 | 老師看不到 |
|---|---|
| 活動成員名單（暱稱、頭像、加入時間） | 學員 email、真實姓名等個資 |
| 綁定本活動的 practice 及其打卡與心得 | 學員自己另外開的任何 practice |
| 活動牆內的留言、回應互動 | 學員在其他活動裡的打卡 |
| 本活動聚合統計（節律熱度、標籤分佈） | 個人足跡、人物誌、追蹤/連結關係 |

### 規則細節

- **綁定發生在加入流程**：學員經邀請連結加入 → 複製活動模板 → 寫入 `cohort_practices`。同意畫面一句話：「你在本活動中建立的這個練習，其打卡內容將對活動發起人可見」。未走此流程的 practice 永不在鏈上。
- **單則打卡可設「僅自己」**（Phase 1 即含，2026-07-16 討論定案）：打卡時一個勾選（預設「活動可見」），資料層一個 boolean。「僅自己」的打卡**計入存在、隱藏內容**——節律熱度圖上該學員仍亮著（老師知道他有在動），但內容不可見。理由：學員知道老師會讀每一則後，最可能的行為是安靜的自我審查（打卡降級為交差文），這在指標上看不見，卻直接腐蝕真實分享與課後轉化。此規則須寫進同意文案。若某班大量打卡轉私密，視為「老師的存在造成壓力」的訊號，是值得回饋給老師的洞察，不是功能故障。
- **與社群可見性正交**：practice 既有的 `privacy_status`/`visibility` 管「對社群」，活動可見性是另一軸，互不覆寫。活動綁定絕不使資料外漏至公共靈感牆。
- **聚合統計走同一條鏈**：任何查詢不允許以 user_id 為起點撈 practice，一律從 `cohort_practices` join。AI 助教（Phase 3）餵給 ai-backend 的資料集同樣受此限制。

### API 三道防線（對照 daodao-server 既有模式）

1. **路由範圍化**：所有老師端 endpoint 掛在 `/api/v1/cohorts/:cohortId/*` 底下，middleware `requireCohortRole('owner')`（比照 `requireOwnership`）。不存在以人為單位的老師端 API。
2. **物件級歸屬檢查（防 IDOR）**：細節頁除驗 owner 外，必驗「該打卡所屬 practice 確實綁定本活動」。測試須明確覆蓋跨活動 ID 替換的 case。
3. **回傳欄位白名單**：Zod response validator 嚴格白名單（成員只回 `id/nickname/avatar/joined_at`）。email 永不給老師——通知學員走通知系統。

### 生命週期

- **學員退出**：綁定解除 → 老師不可見該 practice 一切內容（含歷史的個人視圖）。聚合統計保留已計入的歷史數字；個人可辨識資料消失。已寄出的週報不回收（退出後不再出現在任何**新產出**的視圖與報表）。此取捨寫進同意文案。
- **活動結束**：老師端轉唯讀（可看歷史、可期末匯出），學員後續打卡不再可見。唯讀期 90 天，期滿只留聚合報表——同時是續開下一期的商業鉤子。
- **期末成果匯出**：匯出物離開系統（招生素材），標準高於站內可見——**逐學員 opt-in**，只匯出同意者。
- **共同授課/助教**：可見性跟著 `cohort_members` 的 owner/assistant 角色走，人離開角色即失去存取。MVP 助教與 owner 同權，後續再分層。

## 身分模型：關係制，不加全域角色

不在 RBAC 加 teacher 角色（全域角色無法表達「只能看自己班」的範圍限制，且會產生「有角色沒班級」的殭屍狀態）。改用：

- `cohort_members.role = owner / assistant / member`，權限判斷全部基於 cohort 關係（與既有 `requireOwnership` 模式一致）。
- 開班資格用**配額**控管：每人免費開 1 個小班（15 人內），付費解鎖——比人工審核可規模化，並預埋訂閱鉤子。人工審核只留給「認證講師」標章等信任訊號。

## 功能規劃

### Phase 0 — Concierge 驗證（幾乎零程式碼）

找 3–5 位設計夥伴講師：講師用一般帳號建模板、學員用已上線的複製實踐打卡；「儀表板」由團隊每週人工/script 產出摘要寄講師；期末人工匯出一次。

**招募方式：掛在 GTM w1-w2 執行包的 B 軌 outreach 上跑，不另起爐灶**——「我們人工幫你做全班節律報告一期」本身就是 outreach 信的開信鉤子，比推銷尚不存在的產品誠實，也比訪談更能篩出真的會用的人。

**設計夥伴篩選（GTM 名單標準之上加三條）**：
1. 課程作業天然像 practice（語言每日練習、寫作週更、健身打卡等重複性練習型；一次性大作業型先不要）
2. 一個月內有新班開課（Phase 0 要觀察完整週期）
3. 熱介紹優先（體驗粗糙、人工週報可能遲到，只有關係戶撐得住）

**Go / Kill 訊號（開跑前定案，避免八週後只得到「反應好像不錯」）**：
- **Go**：≥3 位講師走完整期，且 ≥2 位在收到週報後有實際回應行為（去學員打卡下留言/鼓勵）≥2 次——驗證「看見→回應」循環真的發生，而非禮貌性看報告
- **Go（付費意願）**：期末問「下一期 NT$4,900 續嗎」，≥1 位口頭承諾或願意預付折扣價
- **Kill**：學員打卡率與講師上一期（無島島）相比無可感知差異，或多數學員打卡第三週後歸零——陪跑空間本身不成立

基準線的誠實限制：無對照組，「打卡率提升」嚴格說測不出來；n=3–5 階段**質性訪談權重高於數字**——講師願不願意把下一期也搬上來，比任何比率誠實。

### Phase 1 — MVP：開班與看見

| 功能 | 複用積木 |
|---|---|
| 活動（cohort）建立、期間設定、邀請連結/代碼 | 新資料模型 |
| 課程作業模板 → 學員一鍵複製並綁定活動 | 複製實踐已上線，只差綁定 |
| 活動節律儀表板（熱度圖、標籤分佈；守設計原則 1） | user-stats 統計模式可參考 |
| 活動動態牆＋老師快速回應/留言 | 靈感牆、快速回應、留言全已上線，加活動過濾 |
| 打卡「僅自己」勾選（計入存在、隱藏內容） | 打卡流程加一 boolean＋老師端查詢 filter |

### Phase 2 — 陪跑深化與講師交付物

需要關心名單（流動中斷訊號）、活動公告/提醒（複用通知＋觸發式郵件）、**期末批次匯出全班成果頁**（GTM 明列交付物，逐學員 opt-in）、模板置頂/精選、學員個人視圖（限活動範圍）。

### Phase 3 — 商業化與 AI 助教

訂閱/收費（Community 方案 NT$4,900/期；前兩期人工開通＋轉帳，金流工程押後）、AI 助教（活動週報摘要、鼓勵語建議——server 已有 checkin-encouragements）、完課證書（admin 已有 certificates 基礎）。

## 與既有規劃的關係

- **共同挑戰（Challenge）**：結構與 cohort 高度相似（一群人、一段期間、共同主題、群體牆、權限分層）。底層做通用群組容器，`type` 區分 `challenge` / `class`，權限與計費在上層分岔——避免兩套資料模型。
- **Buddy**：獨立（一對一媒合），但同班學員是天然配對池，Phase 2/3 加分項。
- **「期」的語意**：活動有明確起迄，**續開下一期是新 cohort**（複製設定），不是延長——這決定計費週期與匯出邊界。

## 北極星指標

**課程結束後仍留下來自主開 practice 的學員比例**（轉化橋）。輔助：講師續開率、活動期間打卡率相對基準的提升。禁用指標見設計原則 1。
