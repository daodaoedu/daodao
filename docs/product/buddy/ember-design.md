# 火苗（Ember）設計

> Buddy 的「關聯性與承諾感」機制。承接 [`prd.md`](./prd.md) 的「待解的核心設計問題 §2」。
>
> 解決的問題：兩個陌生人如何快速對「這件事」建立關聯與承諾感——讓「沒被完成」這件事，雙方都有感。

---

## 核心假設

> 人不一定會為了自己去完成某件事，但人會因為**不想害別人失敗**，而督促自己去做這件事。

火苗把抽象的「承諾」變成**具體、可見、會變化**的共有物，讓「我的鬆懈會影響到我們之間的東西」這件事被視覺化，使社會承諾感成為持續動力。

---

## 設計決策總覽

| # | 決策 | 結論 |
|---|---|---|
| ① | 歸屬 | **每對 Buddy 共有一簇火苗（1:1）** |
| ② | 燃料 | **打卡是唯一燃料；B 的照看延緩熄滅並累積 B 自己的陪伴值** |
| ③ | 狀態 | 旺 → 微弱 → 將熄 → 餘燼（綁定「距上次打卡幾天」） |
| ④ | 熄滅 | **可逆、不消失、不羞辱**——變餘燼，隨時可重燃 |
| ⑤ | 損失感 | **掛在「共同進度」不在個人**（矛頭指向「我們」） |
| ⑥ | 可見性 | **僅這對 Buddy 兩人可見**，不公開上社群 |
| ⑦ | B 的獎勵 | 獨立「陪伴值」，**不與 A 完成度 / 火苗存活掛勾** |

---

## ① 歸屬：每對 Buddy 共有一簇火苗

- 火苗屬於「一段 Buddy 關係」，不屬於個人、不屬於實踐。
- 一個人有 N 個 Buddy = N 簇火苗。
- **副作用是好的**：火苗多 = 義務多 = 累，這自然限制一個人不該有太多 Buddy，呼應產品基調「少數、低耗能的陪伴」。

理由：要有罪惡感，得有一個具體、指名道姓的「我們」在那裡，而非擴散到一群人。

---

## ② 燃料模型

火苗每天自然降溫（entropy）。在此基礎上：

| 行為 | 對火苗的作用 | 額外效果 |
|---|---|---|
| **A 打卡** | **唯一能加溫、變旺的燃料** | — |
| **B 提醒 / 留言** | **不加溫**，但**延緩熄滅**（拖住 grace period，給 A 回來的機會） | 累積 **B 的陪伴值**（見 ⑦） |
| 對稱情況（兩人都在實踐） | 雙方打卡互為燃料 | — |

**關鍵設計：為什麼提醒不能加溫？**
若 B 光靠狂發提醒就能餵旺火苗，會出現「A 根本沒實踐、火卻很旺」的假象，背離初衷。
**火旺 ⟺ A 真的有在實踐**——這是火苗的誠實性底線。

---

## ③ 狀態與衰退

狀態綁定「距上次打卡的天數」，與既有的守望相助 5 天門檻對齊：

| 狀態 | 觸發條件 | 系統行為 |
|---|---|---|
| 🔥 **旺** | 近期有打卡 | — |
| 🌤 **微弱** | 幾天沒動靜 | — |
| 🥀 **將熄** | 接近 5 天未打卡 | 觸發守望相助，提示 B 去捎訊息 |
| 🌑 **餘燼（沉睡）** | 超過門檻 | 火苗睡著，**不消失** |

> 餘燼不是死亡。火苗永遠不會消失，只會沉睡。

---

## ④ 熄滅可逆（不懲罰、不羞辱）

- 火苗永遠不會「死掉消失」，只會變餘燼／沉睡。
- **任一方一個動作**（A 打卡 或 B 捎訊息）就能**重新點燃**。
- 語氣是**邀請不是指責**：
  - ✅「火苗想你了」
  - ❌「你害火快滅了」

---

## ⑤ 損失感：掛在「共同進度」不在個人

火苗變餘燼時，**會**有損失感——這是動力來源，不能拿掉。但損失的標的是**「我們一起養起來的東西」**，不是懲罰個人：

- 損失標的：**共同連續天數 / 火苗的成長等級**（兩人共建的累積）。
- 變餘燼時，這個共同累積會**明顯下降**（是否完全歸零屬公式細節，見「待後續定義」）。
- 矛頭指向「**我們的火需要重新養起**」，而非「**你失敗了**」。

這樣損失趨避照常運作（人不想失去共建的成果），但：
- 降低個人羞愧 → 守住「允許脆弱」
- 強化「1:1 共有」的結構
- 可重建 → 不變成永久債務

---

## ⑥ 可見性：僅雙方可見

- 火苗狀態**只有這對 Buddy 兩人看得到**，不公開、不上社群動態。
- 對應 PRD 中「守望相助留言公開/私密」待議題——**火苗這一層主張私密**，以守住安全感與允許脆弱。

---

## ⑦ B 的獎勵線（解不對稱問題）

承接 PRD「待解設計問題 §1」：當關係不對稱（B 沒有自己的實踐，純粹陪伴 A）時——

- B 累積獨立的 **「陪伴值 / 陪伴足跡」**，顯示在 B 自己的 profile。
- 來源：**照看次數、收到的感謝**等「B 主動關心」的動作本身。
- **與 A 是否完成、火苗是否存活完全脫鉤**：火苗滅了不是 B 的失敗；B 的價值來自「有沒有去關心」，而非「結果好不好」。

> 這條線確保單向陪伴也有意義，且不會把 A 的失敗連帶懲罰到 B。

---

## 防呆與邊界

- **體面收尾**：一段 Buddy 關係自然結束時，火苗能「好好說再見／封存」，而不是變成一筆永遠未完成的罪惡感債務。
- **避免火苗通膨**：因為每簇火苗都是義務，需要對 Buddy 數量有軟性上限或提示，避免使用者背負過多火苗。

---

## 視覺對應（沿用既有資產與風格）

火苗**不新增吉祥物**，而是錨定在 daodao 既有的視覺世界裡。這降低美術成本、也強化既有世界觀。

### 既有資產盤點（canonical reference）

| 資產 | 路徑 | 角色 |
|---|---|---|
| 小島 | `packages/assets/images/island/nobody-island.gif` | 使用者的世界/化身。**島上已畫有一頂帳篷 + 一簇營火 + 炊煙 + 有臉的太陽** |
| daodao 吉祥物 | `apps/website/.../deco-mascot.svg`、`mascot-face.svg`、`icon/mascot-basic.svg` | 淺水藍圓潤 blob 角色、圓眼、拿放大鏡（好奇/探索的陪伴者）。色：`mascot.aqua #7DD3E3` / `#5CC5E8` |
| 情緒臉 | `packages/assets/images/emotion/*.svg` + `celebrate.json` | 黃色圓角方臉：happy / bored / frustrated / hopeless / fine / neutral + 慶祝動畫 |
| 島嶼狀態變體 | `island/blocked-island.png`、`nobody-island.gif` | 既有的「島況」空狀態畫風 |

### 風格約束（務必遵循）

- **畫風**：扁平插畫（flat illustration）、柔和粉彩、圓潤造型、輕微顆粒紋理、留白充足 —— 對齊 `nobody-island.gif` 與 `deco-mascot.svg`。
- **配色**（取自 `packages/design-tokens/src/colors.ts`）：
  - 火苗暖色 = `logo.orange #F97316` / `logo.yellow #FACC15`（呼應島上既有營火的橘）
  - 水/天空/吉祥物 = `mascot.aqua #7DD3E3`、`primary.base #16B9B3`
  - 餘燼/黯淡 = `gray.mid #9FB5B8` / `gray.light #E4EAE9`

### 對應設計

- **火苗 = 島上既有的營火**：直接複用 `nobody-island.gif` 帳篷旁那簇火，不另造火焰素材。火況越旺，火與光暈越大。
- **共有營火連接兩座島**（呼應 §① 1:1）：成為 Buddy 時，視覺上是「兩座島被一簇共有的營火連起來」，讓「屬於我們的火」**看得見**。
- **炊煙 = 捎個消息**（守望相助）：複用營火的煙，把「捎個消息」做成從一座島升起、飄向對方島的**煙訊（smoke signal）**。
- **daodao 吉祥物 = 陪伴/守望的化身**：那隻拿放大鏡的藍色 blob 就是「在一旁照看火堆的人」。藍（冷）與火（暖）形成對比，象徵 B 的陪伴。B 去關心 A 時由吉祥物現身遞出煙訊；冷色吉祥物守著暖色火堆＝陪伴的視覺隱喻。
- **情緒臉 = 火況的情緒讀數**：複用 `emotion/` 那套，讓島上太陽臉反映火況 —— 旺→happy、幾天沒動→bored、將熄→hopeless；餘燼時呼應既有文案「島上下雨 🌧」。
- **`celebrate.json` = 重燃 / 里程碑動畫**：複用既有慶祝 lottie。
- **餘燼島 = 島嶼狀態變體的新成員**：做一張與 `nobody-island.gif` 同畫風的「營火轉餘燼」島，呼應 §④「不消失、只沉睡」—— 不是死掉的島，是等著被喚醒的島。

## 待後續定義（不阻擋本設計）

- 火苗成長等級 / 共同連續天數的**具體計算公式**（每天降溫多少、打卡加溫多少、grace period 確切天數）。
- 上述視覺對應的**實際美術稿**（火旺 / 微弱 / 將熄 / 餘燼四態、共有營火連島構圖、煙訊動畫），建議進入實作前另開視覺稿。
- 重燃時的動畫與情緒回饋。
- 陪伴值在 profile 上的呈現形式。

---

## 與既有系統的關係

- 不新增通知類型，沿用既有打卡感知 / 守望相助 / 里程碑通知；火苗是這些事件的**視覺化載體**。
- 守望相助的 5 天門檻 = 火苗「將熄」狀態的觸發點，兩者對齊。
