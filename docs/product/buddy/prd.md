# Buddy 功能文件

## 核心概念

Buddy 解決的是「學習孤立感」——不需要深度互動，只要讓使用者知道「有人跟我在做類似的事」就能降低孤立感、提升完成率。

核心價值：**從「只有我」→「有人陪伴」**

Buddy 的連結動機來自**共同目標**（做類似的實踐），不是社交關係。這讓它比傳統交友機制更有學習意義、摩擦更低。

---

## 使用者問題

- 缺乏學習夥伴，難以維持動力
- AI 有其局限，需要真人陪伴
- 不知道去哪裡找「跟我做類似事情的人」
- 練習「真的看見」另一個人——允許脆弱，讓人可以承認自己被影響、而不會被嘲笑

---

## 配對邏輯

Buddy 配對基於**相似**，不要求完全相同：

1. **同模板優先**：使用同一個 `template_id` 建立實踐的人
2. **標題關鍵字相似**：`title` 有相近詞彙的實踐
3. 排除已是 Buddy、已有待處理請求的人

> 實踐是個人建立的，幾乎不可能找到完全相同的 `practice_id`。
> 因此配對應以「類似主題」為基礎，而非精確匹配。

## 功能流程

> 流程分兩階段：**配對**（找到夥伴）和**陪伴**（持續感知彼此）。

### 配對流程

#### 主動邀請

```
用戶在他人的實踐頁面 / Profile 頁面
  → 點「邀請成為 Buddy」              ← 發送 UI（缺前端）
    → 送出請求                         ← 後端已完成
      → 對方收到通知                   ← 通知已完成
        → 接受 / 忽略                  ← 通知頁面已完成
          → 雙方成為 Buddy             ← 後端已完成
```

#### 系統推薦（實踐建立後即觸發）

> **時機調整**：推薦不需等到第一次打卡。實踐**建立完成的當下**就有足夠資訊（`template_id`、`title`）做相似度配對，此時也是用戶對這件事最有動力、最願意找夥伴的時刻。
>
> 因此推薦有兩個觸發點：
> 1. **實踐建立完成後**（主要）——建立成功畫面下方出現推薦卡片
> 2. **打卡成功後**（次要）——尚未配對 Buddy 的用戶，打卡後再次提示

```
用戶完成實踐建立（或打卡）
  → 成功畫面下方出現推薦卡片
    ┌─────────────────────────────┐
    │  還有 3 人也在練習類似的事   │
    │                             │
    │  [頭像] 小明  每天冥想 10分鐘│  邀請
    │  [頭像] 阿華  正念練習      │  邀請
    │  [頭像] Jenny 呼吸練習      │  邀請
    │                             │
    │           暫時跳過          │
    └─────────────────────────────┘
      → 點「邀請」送出請求             ← 後端已完成
        → 對方收到通知                 ← 通知已完成
```

### 陪伴流程

#### 打卡感知——「有人知道我做了」

Buddy 打卡時，對方當天收到最多 1 則聚合通知，不因頻繁打卡被轟炸。

```
Buddy 當天第一次打卡
  → 系統於當天晚間統一聚合同日所有 Buddy 的打卡
    → 推送 1 則 in-app 通知
      「今天有 2 個夥伴在島上出現了」
      ┌────────────────────────────────┐
      │ [頭像] 小明   冥想 10 分鐘     │  👋
      │ [頭像] Jenny  英文口說         │  👋
      └────────────────────────────────┘
        → 點 👋 直接在通知上回應（reaction）
        → 點內容進入對方的打卡詳情頁，可留言
```

#### 守望相助——「有人惦記著你」

Buddy 連續 5 天未打卡，系統提醒你去捎個消息。不是督促，是惦記。

```
Buddy 連續 5 天未打卡
  → 推送 in-app 通知
    「小明 最近沒有出現在島上」
    「去捎個消息？」
      → 點擊進入「傳信」畫面
        ┌────────────────────────────────┐
        │  選一張卡片傳給小明            │
        │                                │
        │  🌊  嘿，還在嗎？              │
        │  ☁️  沒關係，慢慢來            │
        │  🏝️  我在這裡                 │
        │  ✍️  自己寫一句…              │  ← 真人留言
        └────────────────────────────────┘
          → 對方收到 in-app 通知
            「[你的名字] 想到你了」＋卡片內容
              → 點進去看卡片
                → 可選擇回應一張卡片，或直接去打卡
```

> **罐頭訊息會膩**：固定卡片在重複幾次後就失去溫度。要讓「惦記」真實，必須讓 Buddy 能**自己留言提醒**，而非只能選預設卡片。罐頭卡片仍保留為「低門檻起手式」，但要補上自由輸入。
>
> **待討論：留言的公開 / 私密**
> Buddy 的提醒留言應該是**公開**（出現在實踐／打卡頁，他人可見）還是**私密**（只有對方收到）？
> - 公開：互動被看見，可能形成社群氛圍與正向壓力
> - 私密：壓力小、更安全，符合「允許脆弱」原則
> - 可能解法：讓**留言者自行選擇**公開或私密
> 此點尚未拍板，需進一步討論。

#### 慶祝里程碑——「有人為你慶祝」

里程碑定義：Day 7 / Day 30 / Day 100 / 完成整個實踐

```
Buddy 達成里程碑
  → 推送 in-app 通知
    「小明 完成了 30 天的冥想練習 🎉」
    「去留個腳印？」
      → 點擊進入對方的打卡詳情頁（里程碑打卡）
        → 可選 reaction + 選填一句話
          → Buddy 收到慶祝通知
```

#### Buddy 列表頁——「一眼看到所有夥伴狀態」

```
我的夥伴
┌──────────────────────────────────────────┐
│ [頭像] 小明   冥想 10 分鐘   今天在島上 ✓ │
│ [頭像] Jenny  英文口說       3 天未出現 🌧 │
│ [頭像] 阿華   正念練習       Day 30 🎉    │
└──────────────────────────────────────────┘
```

每張卡片顯示：頭像、名字、實踐名稱、最近狀態  
狀態優先級：里程碑 > 今天打卡 > N 天未出現  
點擊卡片 → 進入對方實踐頁

## 設計原則

- **低耗能互動**：不強迫雙方主動溝通，只是讓彼此知道對方存在
- **以實踐為中心**：Buddy 是因為「做類似的事」而連結，不是因為認識
- **觸發時機自然**：打卡後是情緒最正向、最容易想到「有人一起更好」的時刻
- **允許脆弱**：介面設計應讓人感到安全，可以誠實分享進度與困難

---

## 互助網絡的關係

Buddy 是「互助網絡」的核心機制。與其透過填寫 profile 欄位（我能提供什麼 / 我需要什麼），daodao 選擇從**行為推斷**互助配對：

- 你在做某個實踐 = 你在這個領域有經驗或興趣
- 你剛開始某個實踐 = 你可能需要這方面的陪伴

這讓「互助」自然發生，不需要額外的填表成本。

## 待解的核心設計問題

> 以下是目前認為最關鍵、但尚未拍板的設計問題。它們決定 Buddy 能否真正運作，而非只是通知機制的堆疊。

### 1. Buddy 的動機與收獲——不對稱關係怎麼辦？

目前的配對邏輯假設**雙向對稱**：兩人都在做類似實踐，互相陪伴、互相提醒。但實際上可能出現**單向關係**：

> A 需要 Buddy 來提醒、B 來幫忙，但 B 同一時間並沒有進行任何自己的實踐。

這時的問題是：**B 的收獲或獎勵是什麼？**

設計原則：

- **B 的收獲不應與 A 的完成度掛勾。** 否則 B 變成「為 A 的成績負責」，壓力錯置，也讓 A 的失敗連帶懲罰 B。
- B 的價值應來自**「陪伴 / 幫助這個行為本身」**，而不是 A 是否成功。例如：被惦記、被需要、自己的善意被看見、累積屬於自己的「陪伴足跡」。
- **已定**：給 B 一條**獨立於 A 完成度**的回饋線 —— 「陪伴值 / 陪伴足跡」（來源：照看次數、收到的感謝），與 A 是否完成、火苗是否存活完全脫鉤。詳見 [`ember-design.md` §⑦](./ember-design.md)。

### 2. 關聯性與承諾感——兩個陌生人為何在乎彼此？

目前最缺乏的，是**兩個陌生人如何快速對「這件事」建立關聯性**。重點不是他們要多認識彼此，而是：

> 當這件事沒被完成時，他們的感受會是如何？
> A 沒打卡，B 去提醒他——但 **A 為什麼一定要因為被提醒而行動？** 而 **B 又為什麼一定要去提醒？**

換句話說，雙方對彼此的「承諾感」到底有多強？若沒有承諾感，通知與提醒都只是無痛的雜訊。

#### 解法：火苗（Ember）— 共有的承諾載體

設計一個雙方共有的代表物（artifact）—— 一簇**火苗** —— 把抽象的「承諾」變成**具體、可見、會變化**的東西，讓「我的鬆懈會影響到我們之間的東西」被視覺化。

**底層假設：**

> 人不一定會為了自己去完成某件事，但人會因為**不想害別人失敗**，而督促自己去做這件事。

**已定的核心決策**（完整設計見 [`ember-design.md`](./ember-design.md)）：

- **歸屬**：每對 Buddy 共有一簇火苗（1:1）；火苗多 = 義務多，自然限制 Buddy 數量。
- **燃料**：A 打卡是唯一能讓火苗變旺的燃料（火旺 ⟺ A 真的有實踐）；B 的提醒不加溫，但能延緩熄滅、並累積 B 自己的陪伴值。
- **狀態**：旺 → 微弱 → 將熄（對齊守望相助 5 天門檻）→ 餘燼。
- **熄滅可逆**：火苗不會消失，只會變餘燼沉睡，任一方一個動作即可重燃；語氣是邀請（「火苗想你了」）而非指責。
- **損失感掛在「共同進度」**：變餘燼時損失的是兩人共建的累積（共同連續天數 / 成長等級），矛頭指向「我們」而非懲罰個人 —— 兼顧動力與「允許脆弱」。
- **私密**：火苗狀態僅這對 Buddy 兩人可見，不公開上社群。
