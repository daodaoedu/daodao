# 留存診斷：動機、通知系統健檢與唯一工程項

> 版本：v1（2026-07）
> 背景：創辦人診斷「流失的原因是使用者沒有明確要回來使用的強烈動機」。本文件記錄對這個診斷的分析框架、對 daodao-server 通知系統的實際健檢結果，以及結論。
> 結論先講：**動機循環的產品機制幾乎都已建好，缺的是密度（人），不是功能。M1 前唯一值得動的通知工程項只有一個：首個回應即時寄送。**

---

## 1. 診斷框架：動機不能被製造，只能被借用

「用戶沒有強烈動機回來」的錯誤解法是做「讓人想回來的功能」（遊戲化、徽章、每日任務）——那條路終點是靠焦慮驅動的 streak 機器，與島島戒律正面衝突。正確的問題是：**用戶生活裡已經存在的動機，島島能借用哪些、放大哪些？**

| 動機來源 | 機制 | 產品對應 | 現況 |
|----------|------|----------|------|
| 目標的動機 | 「10/26 要考多益」——用戶為自己的目標回來，島島只是目標的容器 | 實踐設定目標日期、顯示「距離考試還有 N 天」（自己的倒數是意義感，不違反戒律；被平台催才是壓力） | ⚠️ 未確認/未建——backlog |
| 社會的動機 | 「被回應」是唯一的 variable reward；回應發生時的站外通知是回訪最強觸發器 | 回應 → notification_events → email digest | ✅ 已建（見 §2），缺水不缺管 |
| 累積的動機 | 記錄越多、離開成本越高——前提是累積看得見（小收成） | 週報統計（本週打卡數/共鳴數/留言數） | ✅ 雛形已建（週日晚間週報） |

**推論一**：沒有明確目標的散客本來就留不住、也不該花力氣留。留存策略的第一步是獲取策略——只收「本來就有理由回來的人」（考期同梯、轉職者）。
**推論二**：三個機制中兩個已建成，流失的主因不是機制缺失，而是機制空轉（見 §3）。

### 已上線的完整鏈路

| 通知偏好/退訂 | `notification-preference.service.ts`、`notification-unsubscribe.service.ts`，未設定預設開啟 | ✅ |
| 即時寄送通道（先例） | `wish.linked` 事件走即時信，不進 digest | ✅ 先例存在 |

### 設計文件

`docs/product/notification/email-notifications.md` ——digest 結構依社交學習價值排序（夥伴動態 → 有人與你互動 → 關注動態 → 共鳴回饋），設計理念（被看見/看見他人/相互問責）與 GTM 策略的動機框架一致。

### 健檢結論

**「有人回應了你但你不知道」的假設不成立**——基礎設施不是瓶頸。管子是好的，但裡面沒有水：零密度 → 打卡沒人回應 → 通知系統無事可通知 → 社會動機的獎勵環節空轉。**留存問題的解法在同梯密度與人肉保底回應（營運），不在通知工程。**

## 3. 唯一建議的工程項：首個回應即時寄送

### Why

新用戶第一張卡收到的第一個回應，是整個留存漏斗的黃金時刻——「有人看見我」的第一次體驗。目前它跟著 4 小時 digest 班次走，情緒 window 早已關閉。此改動把最重要的一次 reward 從批次移到即時。

### 規格草案

- **觸發條件**：`reaction` 或 `comment` 事件，且滿足「收件者的該張 checkin 是其人生第一張打卡」或簡化版「收件者從未收過任何互動通知」
- **行為**：立即寄送單封通知信（複用 `reaction-notification-template.ts` / 留言模板），信內直接展示回應內容與回應者（把 reward 放進信裡，不是「你有新通知」空包彈）
- **去重**：該事件標記為已即時寄送，後續 digest 班次排除（參照 `wish.linked` 的處理模式）
- **尊重偏好**：走既有 `notification_preferences` 檢查
- **範圍外**：不做所有回應即時化（信量與打擾控制，digest 設計是對的），只開「第一次」的特例

### 驗收

新帳號 → 打第一張卡 → 另一帳號回應 → 5 分鐘內收到含回應內容的 email，且該事件不再出現於下一班 digest。

## 4. Backlog（記錄，M1 前不動）

| 項目 | 內容 | 時機 |
|------|------|------|
| 目標日期與倒數 | 實踐支援目標日期欄位，介面顯示「距離目標還有 N 天」——目標動機的產品化，考期同梯的自然配套 | M1 後、第二梯前評估 |
| Email provider 遷移 | 現為 SMTP 帳密寄送（`email.config.ts`，EMAIL_ADDRESS/PASSWORD），量大後有寄送上限與垃圾郵件風險 → 遷移至交易信服務 | M2 後或 digest 日寄送量 >500 封時 |
| 首次打卡 24h 內無回應的兜底 | 若第一張卡 24 小時沒有任何自然回應，通知營運者（人肉保底的觸發器）——比自動假回應誠實 | 第一團開跑時以手動巡檢代替，量大再自動化 |

## 5. 附錄：主題實踐同梯支援度盤點（2026-07-10）

因應同梯策略對主題實踐的功能檢查（C0-2 的落地）：

| 需求 | 現況 | 判定 |
|------|------|------|
| 梯次期間（資料層） | `practices.start_date` / `end_date` 已存在（`410_create_table_practices.sql:18`）；server 驗證允許 7–90 天（`me.validators.ts:249`） | ✅ 不用動 |
| **梯次期間（前端選項）** | 建立實踐 UI 僅四檔 7/14/21/30 天（`create/manual/schema.ts:21`），**無 60/90**；開始日期限未來 14 天內（`step-2.tsx:61`）。現有三活動全 ≤30 天——上限反向塑造了活動形態，90 天考期團與 builder 梯是第一批撞牆者 | 🔴 **缺口 #2（工程量最小）**：`DURATION_DAYS_OPTIONS` 加 60/90 選項＋i18n，server 免改。連帶驗證：複製實踐時期間是否繼承來源——若複製要求重選期間，此修正為入梯硬前置 |
| 複製鏈記錄 | 複製時寫入 `source_practice_id`（`practice.service.ts:1700`） | ✅ 資料一直在記 |
| 打卡引導 | 模板說明欄承載（執行包 Day 2 模板即此設計） | ✅ 不需新欄位 |
| 週題 | 無機制 | ✅ 營運層解決（Discord 公告），第一梯不產品化 |
| **同梯可見性** | `source_practice_id` 僅 admin analytics 讀取，**無任何用戶端出口**（f2e 零引用）——複製同一模板的人彼此隱形 | 🔴 **唯一真缺口** |

**同梯可見性處理**：
1. 第一梯用零工程降級方案——營運者發同梯名單、成員手動互關，靈感牆即可見彼此（30 人內可行）
2. 第一梯跑到一半確認需求後再產品化：實踐頁「同梯區塊」（`WHERE source_practice_id = X` 成員列表＋最新打卡）——先讓手動版證明需求再寫 code

**Backlog**：目標倒數顯示（§4 已列）、兩段式階段結構（先用換打卡引導的營運手法頂）、主辦者節律面板（B 軌 Community 核心交付物，簽下 design partner 後才做——有明確買家才動工）。

## 6. 對 GTM 執行的意義

1. **工程凍結的判斷再次被驗證**：動機三機制中兩個已建好、一個進 backlog，M1 前的通知工程只有 §3 一項。
2. **執行包 Day 4 的「人肉保底回應」升格為留存主策略**——它不是權宜之計，它就是現階段讓通知系統有東西可送的唯一方法。
3. **週報已存在**意味著同梯營運的「同梯週報」可以直接借用現有週報基礎，不用重做。
