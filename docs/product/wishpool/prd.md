# 島島許願池 PRD（Wish Pool / 公開 Roadmap）

版本 v1.3　｜　2026-05-25　｜　參考原型：[好室友 Roadmap 原型](https://housemate-roadmap-prototype.netlify.app/)

> **狀態校準（2026-07-06）：✅ 已上線。** 本文件的需求與設計內容仍有效，但許願池與公開 Roadmap 已實作於程式碼——規劃時請以程式碼為準，勿重做。實作位置：`daodao-server/src/routes/wish.routes.ts`、`daodao-server/src/routes/roadmap.routes.ts`、`daodao-f2e/packages/api/src/services/roadmap.ts`、product app 的 `/roadmap` 頁面。狀態追蹤見 daodao 主 repo 的 `scripts/product_status_manifest.yml` 與 `.claude/skills/product-status-check`。

## Overview

島島阿學在持續迭代的過程中，會從使用者身上收到大量的回饋與功能許願，但目前這些聲音散落在各處（私訊、表單、口頭），既難以彙整，也讓使用者感覺「講了沒有回應」。

「許願池」是一個**公開的產品共創空間**：使用者用低摩擦的引導流程許下對島島的願望，島島團隊把這些願望策展成公開的產品 Roadmap，讓每個人都能看到「我們正在做、正在想的事」，並用投票表達支持。它把單向的客訴／許願，轉化為「使用者與島島一起把產品變更好」的雙向陪伴關係。

本功能採用**兩層資料模型 + 投票**：

- **許願（Wish）**：使用者提交的原始回饋，進入待處理池，只有島島團隊看得到原文。
- **路線圖項目（Roadmap Item）**：團隊從許願池策展、改寫成公開的卡片，帶有狀態（討論中／排程中／已完成…）與支持票數，公開展示在 Roadmap 看板上。
- 使用者可對路線圖項目**投票（支持）**；團隊把相似的許願歸入同一個路線圖項目，票數即代表「有多少人想要這件事」。

> 對照原型的三個組成部分：① 公開 Roadmap 看板（狀態分頁 + 票數 + Hero 統計）、② 三步驟許願流程、③ 兩層策展與投票。本 PRD 涵蓋完整版範圍。

## 已定案決策（v1.3）

- **載體與架構**：整個許願池放在 **product app**，並開一條**公開、免登入的唯讀路由**（例如 `/roadmap`）。未登入/未註冊者可瀏覽完整看板與分享；許願 wizard、投票與完成頁皆在同一 app 內原生完成，不跨 app。需將 `/roadmap` 於全站登入 guard 設為公開白名單。官網 website 日後可放入口連結導向，不搬移功能。
- **登入與訪客體驗**：觀看完全開放且「**有引導、不空白**」；**許願與投票需登入**，於**互動當下**引導登入/註冊（說明價值），登入後**返回完成原動作**。未登入時凡需登入才有資料的區塊，以引導內容（價值說明 + 範例 + 註冊 CTA）呈現（詳見 §7）。
- **聯絡方式**：許願者一律綁定帳號，進度通知走帳號（站內 + Email），不向匿名訪客收 email。
- **許願分類（#1）**：前台維持**粗八類**低摩擦選擇；團隊於後台策展時補**內部模組子標籤**做精準歸類。
- **Hero 統計（#2）**：參與夥伴數 = 曾許願或投票的**去重使用者數**；累積回饋數 = **許願總則數**；以「N+」呈現，**每小時快取**，不求即時。
- **趨勢報表（#3）**：後台（admin-ui）提供許願/投票的彙整與趨勢報表（如本月最多許願分類、熱門項目排行）。

## Objectives & Goals

- **讓回饋被看見、被回應**：使用者許願後能在公開 Roadmap 看到進度，消除「許願石沉大海」的挫折感，提升回饋意願。
- **降低許願摩擦**：用「選分類 → 舉例子 → 說期待」的三步驟引導，讓不擅長寫需求的使用者也能輕鬆表達。
- **建立產品共創感**：用「島島正在進化，而你是其中一份力量」的敘事，把使用者拉進產品旅程，強化社群認同與留存。
- **幫團隊排優先序**：透過投票與許願歸併量化「呼聲」，作為 Roadmap 排序的客觀參考。
- **沉澱可量化的回饋資產**：原始許願結構化入庫（分類／情境／期待），可被搜尋、彙整、追蹤趨勢。

## Personas & User Stories

| 角色 | 用戶故事（User Story） |
| :---- | :---- |
| **熱心封測夥伴** | 身為一位常用島島的封測夥伴，我希望我提的建議能被看見、被回應，知道島島真的有在聽，這樣我才願意繼續給回饋。 |
| **沉默觀察者** | 身為一位不擅長寫長需求的使用者，我希望能對別人已經提過的願望直接「+1」投票，不用自己重寫一遍。 |
| **方向感需求者** | 身為一位想知道島島接下來要做什麼的使用者，我希望能看到一個公開的 Roadmap，知道哪些功能在排程中、哪些還在討論。 |
| **島島產品團隊（管理者）** | 身為產品團隊，我希望能在後台快速分流許願、把相似的歸成一個項目、改寫成對外卡片並設定狀態，讓 Roadmap 隨時反映真實進度。 |

| 名詞 | 說明 |
| :---- | :---- |

## Features and Functionalities

### 1. 公開 Roadmap 看板（Roadmap Board）

許願池的主頁，放在 **product app 的公開免登入唯讀路由**（例如 `/roadmap`），任何人（含未登入、未註冊者）皆可瀏覽完整看板與分享。對應原型首頁。未登入者看到的是**完整的 Roadmap 內容**（非登入牆、非空白頁）；看板上的互動按鈕（許願、投票）對未登入者點擊時才引導登入（見 §2、§3、§7）。

- **Hero 區**：標語「**島島正在進化，而你是其中一份力量**」＋副標「以下是我們正在做、正在想的事」，並顯示動態統計（見 §6）。
- **狀態分頁（Tabs）**：
  | 分頁 | 對應狀態 | 說明 |
  | :---- | :---- | :---- |
  | 全部 | 所有公開項目 | 預設分頁，依票數／置頂排序 |
  | 排程中 | `planned` + `in_progress` | 已確定要做或正在做 |
  | 討論中 | `discussing` | 團隊正在評估、想清楚做法 |
  | 已完成 | `done` | 已上線，兌現「進度會更新在這裡」的承諾 |
- **路線圖卡片**：每張卡顯示標題（痛點，一句話）、描述（我們打算怎麼做／正在想什麼）、分類標籤、**支持票數（❤️ + 數字）與投票按鈕**。
- **排序**：預設依「票數 desc → 更新時間 desc」；團隊可手動置頂（pinned）少數項目。
- **載入更多**：超過首屏數量時以「展開更多（共 N 則）」漸進載入（對照原型）。
- **許願 CTA 區**：底部固定區塊「App 上我想要的功能都沒有？什麼都可以說，我們來者不拒」＋「點我許願」按鈕，開啟許願流程（§2）。
- **SEO / 分享 meta**：路由公開，需以 Next.js metadata 輸出靜態 meta——標題「島島 Roadmap｜我們正在做、正在想的事」、描述「看看島島接下來要做什麼，也許下你的願望，一起把島島變更好」、OG image 先用固定品牌圖（logo + 「島島正在進化」），確保社群分享有好預覽、爬蟲可索引。動態帶統計數字的 OG image 為日後選項。

### 2. 三步驟許願流程（Wish Wizard）

點「點我許願」後開啟引導式 Modal，對應原型的 Step 1–3 + 確認 + 完成。**許願需登入**：未登入者點「點我許願」時，引導登入/註冊，登入後**返回**繼續許願流程（不遺失已填內容為佳）。

- **Step 1 / 3 — 哪裡讓你不順？**（選分類）
  - 單選一個分類。前台維持**粗八類**（低摩擦、不易選錯、功能改版不需動清單），最終文案：
    1. 操作與流程（不順手、找不到、卡關）
    2. 主題實踐與打卡
    3. 社群互動（留言、反應、關注、連結）
    4. 探索與資源（找內容／靈感／資源）
    5. 共同挑戰
    6. AI 與推薦
    7. 個人檔案與通知
    8. 其他
  - 分類為**必填**，選完可進下一步。
  - 精準歸類由團隊在後台策展時以**內部模組子標籤**補上（前台不顯示，見 §4）；選「其他」的許願於策展時**強制**補子標籤，避免「其他」變垃圾桶，並作為「需新增分類」的訊號。
- **Step 2 / 3 — 可以舉個例子嗎？**（描述情境）
  - 自由輸入：「什麼時候、在做什麼、發生了什麼事」。
  - 建議必填（可設最小字數，例如 ≥ 10 字），但提供「想不到也沒關係」的寬鬆語氣。
- **Step 3 / 3 — 你希望怎樣？**（描述期待）
  - 自由輸入期待，文案強調「不用管做不做得到」，鼓勵天馬行空。
- **確認與聯絡方式**：
  - 許願者一律為登入使用者，自動帶入身分。
  - 進度通知預設送至帳號 email，使用者可選填覆寫其他聯絡 email（對應原型「留下你的 email，讓我們保持聯繫」，但已非必填）。
  - 顯示許願內容摘要供確認後「確認送出」。
- **完成頁**：「已收到！謝謝你對島島的支持，每一則許願我們都會認真對待，產品進度會更新在這裡」＋「看看所有許願」回到 Roadmap 看板。
- **導覽**：步驟間可「上一步／下一步」，未填必填欄位時禁止前進；流程進度條顯示 Step X / 3。

### 3. 投票（支持）機制

- 使用者可對任一公開路線圖項目投票（❤️ 切換），代表「我也想要」。
- **一人一項目一票**，可取消（toggle，冪等）。
- **登入要求**：投票需登入以確保去重的完整性。未登入點擊投票時引導登入/註冊，登入後**返回並完成該次投票**，不走死路。
- 卡片即時顯示票數變化（樂觀更新）。
- **歸併即投票**：團隊把一則許願掛到某項目時，系統視為該許願者（若可辨識身分）對此項目投票一次，使票數自然反映「許願 + 主動投票」的總呼聲。

### 4. 兩層策展與狀態流轉（後台 / daodao-admin-ui）

供島島團隊使用，是兩層模型的核心。

- **許願收件匣（Wish Inbox）**：
  - 列出 `pending` 許願，可依分類篩選、關鍵字搜尋、依時間排序。
  - 顯示每則許願的分類／情境／期待／來源／聯絡方式。
  - 操作：
    - **歸入現有項目**：選一個路線圖項目，將此許願 link 過去（自動 +1 票），許願狀態→ `linked`。
    - **建立新路線圖項目**：以此許願為基礎，填寫對外標題與描述、選分類、設初始狀態，許願狀態→ `linked`。
    - **封存**：重複／無效／不適用，許願狀態→ `archived`（保留原文不刪除）。
  - 策展時可為許願補上**內部模組子標籤**（精準歸類，前台不顯示）；分類為「其他」的許願**強制**補子標籤。
- **路線圖項目管理**：
  - 新增／編輯卡片的標題、描述、分類。
  - 設定狀態（見 §狀態機）；標記「已完成」時記錄 `shipped_at` 並觸發進度通知（§5）。
  - 設定是否公開（`is_public`）、置頂（pinned）、排序。
  - 檢視該項目目前歸入了哪些許願、票數來源。
- **趨勢與彙整報表**：admin-ui 一頁 dashboard（預設本月、可切時間範圍），先做四個決策用指標：① 本月最多許願的分類（排行）、② 熱門路線圖項目排行（依票數，可篩狀態）、③ 許願量趨勢（週/月折線，可依分類拆）、④ 待處理池積壓（pending 數、最舊一筆等待時間）。其餘指標有需要再加，不過度建。

### 5. 進度回饋與通知

兌現原型「產品進度會更新在這裡」「讓我們保持聯繫」的承諾。

- **許願被採納**：當某使用者的許願被歸入或促成一個路線圖項目時，通知該使用者「你的許願被收進 Roadmap 了」。
- **項目進度更新**：當使用者投票／許願所屬的項目狀態推進（尤其→ `done`）時，通知所有支持者。
- **管道**：站內通知（沿用既有 notification system）＋ Email（取帳號 email 或使用者覆寫的聯絡 email，沿用既有 Email Worker）。
- 通知頻率需聚合，避免狀態頻繁變動造成轟炸（同一項目短時間多次變動只發一次）。

### 6. Hero 統計（Social Proof）

- 顯示兩個動態數字（對照原型「178+ 封測夥伴 / 106+ 產品回饋」）：
  - **參與夥伴數**：曾許願或投票的去重使用者數。
  - **累積回饋數**：許願總則數。
- 數字以「N+」呈現，**每小時快取**更新一次，不要求即時精確。

### 7. 未登入訪客體驗與登入引導（Guest Experience）

許願池是未註冊者認識島島的重要入口，未登入體驗必須「**有引導、不空白**」。

- **首頁/看板不擋登入**：公開唯讀路由，未登入即可看到完整的 Roadmap 內容（不是登入牆、不是空白頁），讓人因「看見島島正在做的事」而理解產品。
- **互動才要登入**：點許願或投票時才彈出登入引導，且引導需說明「為什麼登入、登入後能做什麼」（例如：許願會被團隊認真看、可追蹤進度），並提供 Google 一鍵等低摩擦選項；登入後**返回完成原動作**。
- **需登入才有資料的區塊不得空白**：凡依賴登入才有內容的區塊（如「我的許願」「我支持的項目」「我的通知」），未登入時 SHALL 顯示**引導內容**——價值說明 + 範例示意 + 註冊 CTA，而非空狀態或單純「請登入」。
- **語氣**：沿用島島既有 onboarding 的溫暖引導語氣，把訪客當「未來的夥伴」邀請，而非擋在門外。

## 資料模型（Data Model）

採兩層 + 投票，三張主表。此處為 PRD 層級概念模型，實際欄位／型別／migration 於 FRD 或實作時細化。

### `wishes`（許願：原始回饋，僅後台可見）

| 欄位 | 說明 |
| :---- | :---- |
| `id` | 主鍵 |
| `category` | 分類 enum（§2 八類） |
| `situation` | Step 2 情境描述 |
| `desire` | Step 3 期待描述 |
| `submitter_user_id` | 許願者（必填；許願需登入） |
| `email` | 進度通知 email（預設取帳號 email，可選填覆寫） |
| `status` | `pending` / `linked` / `archived` |
| `linked_roadmap_item_id` | 歸入的路線圖項目（null 表示尚未策展） |
| `internal_module_tag` | 內部模組子標籤（團隊策展時填，精準歸類，前台不顯示，可為 null） |
| `source` | `web` / `app` |
| `created_at` / `updated_at` | 時間戳 |

### `roadmap_items`（路線圖項目：對外公開卡片）

| 欄位 | 說明 |
| :---- | :---- |
| `id` | 主鍵 |
| `title` | 對外標題（痛點，一句話） |
| `description` | 對外描述（我們打算怎麼做 / 正在想什麼） |
| `category` | 分類 enum |
| `status` | `collected` / `discussing` / `planned` / `in_progress` / `done` / `parked` |
| `support_count` | 去正規化的支持票數（= 去重支持者數） |
| `is_public` | 是否公開於看板（`parked`／內部項目可隱藏） |
| `pinned` / `sort_order` | 置頂與排序 |
| `shipped_at` | 完成時間（status=done 時設定） |
| `created_at` / `updated_at` | 時間戳 |

### `roadmap_item_supports`（投票 / 支持）

| 欄位 | 說明 |
| :---- | :---- |
| `id` | 主鍵 |
| `roadmap_item_id` | 對應項目 |
| `user_id` | 支持者（去重鍵） |
| `origin` | `vote`（主動投票）/ `wish_link`（由許願歸併產生） |
| `created_at` | 時間戳 |
| — | UNIQUE(`roadmap_item_id`, `user_id`) |

> `support_count` 由 `roadmap_item_supports` 去重計數維護（投票與歸併共用同一張表，使「許願者」與「投票者」統一計數，避免重複灌票）。

## 狀態機（Status Lifecycle）

### 許願 Wish

```
pending ──(歸入/促成項目)──► linked
   └──────────(重複/無效)──► archived
```

### 路線圖項目 Roadmap Item

```
collected（許願中，剛建立）
   ├──► discussing（討論中）
   │        └──► planned（排程中）──► in_progress（進行中）──► done（已完成，記 shipped_at）
   └──► parked（暫不考慮，預設不公開或收進「考慮過」區）
```

- 看板分頁與狀態對照見 §1。
- 狀態允許回退（例如 planned → discussing），由團隊決定；任何狀態變更皆寫入更新時間。

## UX Notes

- **低摩擦優先**：許願 Wizard 每步只問一件事、文案口語化、用寬鬆語氣（「簡單說就好」「想不到也沒關係」「不用管做不做得到」），降低空白焦慮。
- **可見的回應感**：完成頁與通知都要強調「會更新在這裡」，並提供一鍵回看板的入口，讓使用者感受到閉環。
- **訪客友善 + 拉新漏斗**：公開看板對未登入者（含未註冊者）開放瀏覽與分享，先建立「島島持續進化」的信任；互動才引導登入，且未登入的各區塊一律「**有引導、不空白**」（詳見 §7）。登入後返回完成原動作，登入牆即自然的註冊 CTA。
- **保留已填內容**：未登入者填到一半（或填完）才被要求登入時，登入後應還原已填內容，降低流失。實作以**前端 localStorage 草稿**為主（含 24h 過期），登入返回後回填 wizard、送出成功即清除；後端 draft 留給「跨裝置續寫」等進階需求，現階段不做。
- **避免重複許願**：許願前可在 Step 1/2 帶出「相似的願望可能已存在」的提示，引導使用者直接去看板投票，減少重複池量（可列為次階段優化）。
- **內容安全**：許願與對外卡片之間隔著人工策展，原始許願不直接公開，避免不當內容或個資外洩；對外文案皆由團隊改寫。
- **品牌語氣**：沿用島島溫暖、陪伴的語氣，Hero 與 CTA 文案以「一起把島島變更好」為核心敘事。

## Non-goals

- **不做即時公開的原始許願牆**：使用者送出的原文不直接對外顯示，一律經團隊策展成路線圖項目後才公開（兩層模型的核心）。
- **不做使用者之間的留言／討論串**：許願池聚焦「許願 + 投票 + 看進度」，不在此引入評論系統（既有社群互動功能另有規劃）。
- **不做需求的工時估算、指派、看板拖拉等專案管理功能**：對外只呈現狀態，內部排程細節不公開。
- **不做匿名許願/投票**：許願與投票一律需登入；未登入僅能瀏覽。不向匿名訪客收 email。

## Release Criteria / 驗收標準

- **功能完整性**：三步驟許願可成功送出並落地 `wishes`；看板四個分頁正確依狀態過濾；投票可切換且票數正確去重。
- **兩層流轉**：後台可將許願歸併／促成為路線圖項目、設定狀態；歸併後票數正確 +1，許願狀態轉為 `linked`。
- **進度回饋**：項目標記 `done` 時，支持者（含留 email 的訪客）收到通知。
- **效能**：看板首屏查詢 p95 < 500ms；投票切換延遲 < 200ms；Hero 統計可快取。
- **可用性**：未登入可瀏覽看板；點許願/投票時登入引導順暢，登入後返回完成原動作，無死路。
- **安全**：原始許願不外洩；對外卡片皆經策展；API 對輸入做 Zod 驗證。
