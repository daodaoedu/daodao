## 分期開發計畫

### Phase 1：讓已有的東西完整可用

**目標：** 最高 ROI — AI 洞察已在 DB 但使用者看不到，接上就有價值。

#### 1a. AI 洞察搬上總結頁

- `PracticeSummary` type 加 `insight?: string`
- `getPracticeSummary()` 從 practice 資料帶出 insight
- summary page 新增「AI 洞察」區塊
- 有 insight → 顯示完整內容
- 無 insight → 顯示鼓勵語
- 開發量：小（前端改 3 個檔案）

#### 1b. Public View（公開分享頁）

- 新增 `/practices/[id]/showcase` 頁面（訪客可見）
- 顯示：卡片 + AI 洞察 + 精選打卡
- 不顯示：匯出、覆盤、編輯功能
- Owner view 增加「預覽公開頁」+ 一鍵複製連結
- 權限：`privacy_status === "public"` 時任何人可見

#### 1c. .txt 匯出 + AI Prompt 複製

- 按匯出 FRD 原始 scope 實作
- 雙入口：底部按鈕 + kebab menu
- Owner-only
- 零後端依賴，純前端字串組裝 + Blob 下載
- 檔案格式見「匯出 FRD」的排版規格

**預估：** 1-2 週
**依賴：** 無

---

### Phase 2：增加深度

**目標：** 讓完成實踐不只是「結束」，而是有深度的回顧和社群認可。

#### 2a. 覆盤 UI

- DB 的 `reflection` 欄位已存在
- AI backend `Practice` model 需補 `reflection` column
- daodao-server 需要 PATCH endpoint 更新 reflection
- 前端：completion 觸發時提示填寫覆盤
- 提供 ORID + 簡單回顧兩種模板
- 完成後覆盤置頂 72 小時
- **覆盤跟 AI 洞察並聯**（各自獨立，不互 feed）
  - 原因：降低耦合，可獨立開發和上線
  - 未來可選擇串聯（AI 二次合成覆盤 + 洞察）

#### 2b. AI 觸發品質門檻

- `insight_scheduler.py` 的 `get_pending_practices()` 加條件：
  - `progress_percentage >= 80`
  - 平均打卡字數 >= 50（需 subquery 計算）
- 未達標 → insight 留 null
- 前端已在 Phase 1a 處理了 null 的顯示（鼓勵語）

#### 2c. 解析度%（可選遊戲化）

- 純前端計算：`clarity% = min(字數% / 50, 進度% / 80)`
- 實踐進行中：在「實踐洞察生成中」入口顯示 clarity%
- 未到結束日：上限鎖 99%
- 到達結束日 + 兩項指標達標：解鎖 100%

#### 2d. 見證指標

- DB 表 `practice_social_proofs` 已存在
- 後端 service 已在 worktree（需 merge + 補 API endpoints）
- 新增：`POST /api/v1/practices/:id/social-proof`（toggle）
- 新增：`GET /api/v1/practices/:id/social-proof/status`
- 前端：Insightful / Referenced / Witnessed 按鈕
- MVP：主動點擊觸發（不用滾動偵測）

**預估：** 3-4 週
**依賴：** Phase 1 完成

---

### Phase 3：PDF 作品集（信用資產化）

**目標：** 從 .txt 升級為有品牌力的可攜帶數位資產。

詳細規劃見 `openspec/changes/practice-journey-export/`。

核心調整：
- 此 change 延後至 Phase 2 完成後再啟動
- 原因：PDF 需要 AI 洞察 + 覆盤 + 見證數據才完整，且見證需要上線一段時間累積數據
- 驗證章條件（持續 ≥ 14 天 + Insightful ≥ 3）在見證功能上線初期形同虛設

**預估：** 2-3 週
**依賴：** Phase 2 完成 + 見證數據累積

## 技術決策記錄

| 決策 | 選擇 | 原因 |
|------|------|------|
| `raw_logs_accumulator` | 不使用 | AI backend 已用更好的做法（直接 query + 統計），accumulator 有一致性風險且無實際效能需求 |
| 覆盤 vs AI 洞察關係 | 並聯（各自獨立） | 降低耦合，可獨立開發上線；未來可選擇串聯 |
| AI 觸發門檻 | Phase 2 才加 | Phase 1 先讓所有 completed practice 的 insight 可見，快速驗證價值 |
| 匯出分期 | .txt 先（Phase 1）→ PDF 後（Phase 3） | .txt 零依賴可立刻做；PDF 需要覆盤 + 見證數據 |
| Witnessed 觸發方式 | MVP 主動按鈕 | 技術最簡，避免定義「閱讀完成」的模糊地帶 |
| PDF 渲染 | `@react-pdf/renderer` | 前端處理，文字可選取，支援分頁（Phase 3 決策，沿用 OpenSpec） |
