# 通知觸發與發送引擎

## **Purpose** 

定義通知系統的後端觸發邏輯、頻道路由規則及發送效率，確保「即時性」與「聚合邏輯」能有效運作，以提升用戶回流率並減少數位疲勞。

## **Scope**

涵蓋後端事件監聽、設定頁面 UI、通知偏好管理、以及 Email 取消訂閱機制。

## **Functional Requirements** 

### **通知類型**

* N01: Email 寄送至使用者 email   
* N02: In-App Notification 網站或App 內鈴鐺提示通知  
* N03: In-App Push App 彈跳通知

### **事件監聽與觸發 (Event Triggers)**

系統需監聽以下事件並根據預設頻道發送通知：

* **反應 (Reaction)**：當用戶內容被按讚(快速回應)時觸發。  
* **留言 (Comment)**：當有人在用戶內容下留言、被回覆或使用者被@時觸發。  
* **關注 (Follow)**：當「自己」或「主題實踐」被關注。  
* **連結請求 (Connect)**：當收到建立夥伴關係請求時觸發，需包含「連結請求」內容。  
* **連結確認**：當請求被接受時觸發。  
* **關注的主題實踐更新**：打卡、結束  
* **關注的人更新**：開始一個主題實踐或請求 Buddy

### **分項控制 (Granular Controls)**

* V: 沒有設定選項，一律通知  
* ロ: 用戶可勾選，Default 不勾  
* ☑: 用戶可勾選，Default 勾選

| 類型 | N01 | N02 | N03 |
| :---- | :---: | :---: | :---: |
| 反應 (Reaction) | ☑ | V | ☑ |
| 留言 (Comment) and @ | ☑ | V | ☑ |
| 關注 (Follow) | ☑ | V | ☑ |
| 連結請求 (Connect) | ☑ | V | ☑ |
| 當請求被接受時 | ☑ | V | ☑ |
| 關注的主題實踐更新 | ☑ | V | ☑ |
| 關注的人開始一個主題實踐 | ☑ | V | ☑ |
| 關注的人請求 Buddy | ☑ | V | ☑ |
| 週報 | ☑ | N/A | N/A |

### **Email 聚合與摘要 (Email Aggregation)**

* 對於非即時性的 Email 需求，系統需支持「學習成果摘要」功能，將一週內的反應、關注數量統計後，於預定時間（如每週日）發送一封摘要郵件。

### **Email 退訂機制 (Unsubscribe)**

* 每一封通知 Email 的頁尾（Footer）必須包含：  
  * `取消訂閱此類通知` 的直達連結。  
  * `前往通知設定` 的連結。  
* 點擊取消訂閱後，系統應立即更新資料庫，下一封同類郵件不得發送。

# 感應發送邏輯

## **Purpose** 

重新定義通知的「到達時機」，從「事件驅動」轉向「情境驅動」。針對 MVP 階段優化開發成本，透過固定週期的「頻道分流」，確保通知能作為「有意義的檢核點」，同時建立用戶回流的節奏感。

## **2\. Scope** 

包含通知發送的優先級演算法、安靜時間管理、Web In-app 顯示邏輯、以及「P1+P2 混合型」Email 聚合發送服務。

## **3\. Functional Requirements** 

### **優先級別與時機定義 (Priority Framework)**

系統需根據事件性質將通知分為三類：

* P1 (高價值)：每1小時整點更新一次  
* P2 (鼓勵性)：每1小時整點更新一次  
* P3 (學習摘要)： 週期性 Email

| 類型 | P1 | P2 | P3 |
| :---- | :---: | :---: | :---: |
| 反應 (Reaction) | N/A | V | N/A |
| 留言 (Comment) and @ | V | N/A | N/A |
| 關注 (Follow) | N/A | V | N/A |
| 連結請求 (Connect) | V | N/A | N/A |
| 當請求被接受時 | V | N/A | N/A |
| 關注/連結的人開始一個主題實踐 | V | N/A | N/A |
| 關注/連結的人請求 Buddy | V | N/A | N/A |
| 週報 | N/A | N/A | V |

### **優先級別與雙頻道發送邏輯 (MVP Framework)**

系統將社交信號 (P1 & P2) 在 Web 端採取統一的批次節奏，但在 Email 端進行更長週期的聚合，以適應 Web-only 的使用習慣：

* **Web In-app 顯示邏輯 (每1小時批次)**  
  * **類別**：包含 P1 (高價值：連結請求、訊息回覆) 與 P2 (鼓勵性：反應、關注)。  
  * **邏輯**：系統每小時整點更新一次通知中心（例如 10:00, 11:00）。  
  * **設計考量**：統一為每1小時能降低後端複雜度，且用戶僅在進入/刷新網頁時才會感知到更新，噪音風險極低。

* **Email 整合發送邏輯 (4小時週期)**  
  * **類別**：包含所有 P1 與 P2 事件。  
  * **邏輯**：系統每四小時整點（如 08:00, 12:00, 16:00, 20:00）發送一封「互動摘要」。  
  * **MVP 簡化原則**：**不進行已讀狀態檢查**。即使兩小時前用戶已在 Web 端讀過該訊息，該訊息仍會出現在下一個 Email 批次中，作為社交存檔與回流觸發。

* **P3：學習足跡摘要 (週期性 Email)**  
  * **類別**：本週進度、探索的新足跡匯總。  
  * **邏輯**：僅在週日晚間或週一晨間發送。

### **內容呈現與排序邏輯 (Content Aggregation)**

* **Web UI 排序**：雖然 P1 與 P2 同時更新，但在通知中心列表中，**P1 事件必須置頂**於 P2 事件之上，確保「對話」優先於「按讚」。  
* **Email 結構**：Email 內容需明確區分「重要連結與討論 (P1)」與「共鳴回饋 (P2)」兩大區塊。

## **Test Points** 

* \[ \] **P1/P2 Web 更新頻率測試**：驗證 Web 通知中心是否分別按 1 小時頻率正確刷新。  
* \[ \] **Email 混合發送測試**：驗證 09:15 的「回覆」與 11:30 的「按讚」，是否統一在 12:00 的 Email 摘要中出現。  
* \[ \] **無過濾邏輯測試**：驗證用戶在 11:50 於 Web 已讀某通知，12:00 的 Email 是否仍包含該通知。  
* \[ \] **安靜時間批次測試**：驗證凌晨時段產生的所有 P1、P2 事件，是否都在 08:00 的首封 Email 中被一次性彙整發送。
