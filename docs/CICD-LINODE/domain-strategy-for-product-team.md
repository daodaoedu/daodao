## 為什麼需要思考這個問題？

### 核心問題

當島島同時需要：
- 📢 **官網**：向新訪客介紹我們是誰、做什麼、為什麼選擇我們
- 🎯 **產品**：讓用戶登入後使用平台功能

就需要決定：**這兩者應該放在什麼網址？**

### 這個決定會影響

| 影響層面 | 具體影響 |
|---------|---------|
| 🎨 **用戶體驗** | 訪客如何找到我們？註冊後如何進入產品？ |
| 📊 **SEO 表現** | Google 搜尋結果中的排名 |
| 🎯 **轉換率** | 訪客變成用戶的比例 |
| 💼 **品牌形象** | 專業度、可信度 |
| 📈 **營運效率** | 內容更新、A/B 測試的便利性 |
| 🔧 **開發成本** | 需要維護幾個網站？ |

### 一個類比

想像你經營一家實體店：

```
方案 A（官網 + 產品分離）：
├── 門市展示空間（官網）      → 吸引路人、展示產品
└── 會員服務區（產品）        → 會員專屬功能

方案 B（官網 + 產品合併）：
└── 開放式空間                → 路人可以逛，會員可以使用更多功能
```

## 三種主流做法對比

### 方案 A：官網和產品用不同網址 ⭐ **最常見（55%）**

#### 域名設計

```
官網：https://daodao.so
      ↓（點擊「開始使用」）
產品：https://app.daodao.so
```

#### 視覺化流程

```
┌─────────────────────────────────────────┐
│         新訪客打開 daodao.so            │
│                                         │
│   ┌─────────────────────────────┐     │
│   │  精美的首頁                 │     │
│   │  - 大標題：我們的價值主張  │     │
│   │  - 功能介紹卡片            │     │
│   │  - 用戶評價                │     │
│   │  - [開始使用] 按鈕 ◄────┐ │     │
│   └─────────────────────────────┘  │ │     │
│                                    │ │     │
└────────────────────────────────────┼─┘     │
                                     │       │
                  點擊按鈕後跳轉      │       │
                                     ▼       │
┌─────────────────────────────────────────┐ │
│      跳轉到 app.daodao.so               │ │
│                                         │ │
│   ┌─────────────────────────────┐     │ │
│   │  註冊頁面                   │     │ │
│   │  - Email                    │     │ │
│   │  - 密碼                     │     │ │
│   │  - [註冊] 按鈕              │     │ │
│   └─────────────────────────────┘     │ │
│                                         │ │
└─────────────────────────────────────────┘ │
                                             │
            註冊後停留在 app.daodao.so ◄─────┘
```

#### 用戶體驗特點

**新訪客（第一次來）**
1. 輸入 `daodao.so` 或從 Google 點進來
2. 看到專業的官網介紹
3. 被吸引後點擊「開始使用」
4. 跳轉到 `app.daodao.so` 註冊
5. 以後都在 `app.daodao.so` 使用

**已註冊用戶（常客）**
1. 直接訪問 `app.daodao.so`
2. 自動登入
3. 進入產品

#### 優點（從產品角度）

✅ **轉換漏斗清晰**
```
認知 → 興趣 → 註冊 → 使用
(官網)          (產品)
```
可以在每個階段精準優化

✅ **內容更新靈活**
- 官網改版不影響產品
- 可以快速測試不同的文案和設計
- 節慶活動、促銷可以只改官網

✅ **數據分析清晰**
- Google Analytics：官網看訪客行為
- 產品分析工具：產品看用戶行為
- 不會混淆

✅ **品牌專業度**
- 符合主流 SaaS 產品做法
- 給人「正規軍」的感覺
- 投資人、合作夥伴會覺得專業

#### 缺點

⚠️ **需要維護兩個網站**
- 設計風格要保持一致
- 導航邏輯要統一
- 增加維護成本

⚠️ **域名跳轉**
- 用戶會感覺「跳到另一個網站」
- 可能有短暫的遲疑（但已經是業界常態）

#### 誰在用這個方案？

**國際知名產品**：
- **Slack** - 團隊協作工具
- **Notion** - 筆記與協作
- **Asana** - 專案管理
- **Dropbox** - 雲端儲存
- **Monday.com** - 工作管理
- **Linear** - 專案追蹤
- **Figma** - 設計工具（部分功能）

### 方案 B：官網和產品用同一個網址 🔄 **第二常見（30%）**

#### 域名設計

```
https://daodao.so        → 未登入看官網，已登入看產品
https://daodao.so/login  → 登入頁
```

#### 視覺化流程

```
┌─────────────────────────────────────────┐
│    訪客打開 daodao.so（未登入）         │
│                                         │
│   ┌─────────────────────────────┐     │
│   │  條件式首頁                 │     │
│   │  偵測：「沒有登入」         │     │
│   │  顯示：官網內容             │     │
│   │  - 產品介紹                │     │
│   │  - [註冊] [登入] 按鈕      │     │
│   └─────────────────────────────┘     │
└─────────────────────────────────────────┘
                   │
                   │ 註冊/登入後
                   ▼
┌─────────────────────────────────────────┐
│    用戶打開 daodao.so（已登入）         │
│                                         │
│   ┌─────────────────────────────┐     │
│   │  條件式首頁                 │     │
│   │  偵測：「已登入」           │     │
│   │  自動跳轉到：產品儀表板     │     │
│   │  或直接顯示產品內容         │     │
│   └─────────────────────────────┘     │
└─────────────────────────────────────────┘
```

#### 用戶體驗特點

**新訪客**
1. 輸入 `daodao.so`
2. 看到官網介紹
3. 點擊「註冊」
4. 還是在 `daodao.so`，只是網址變成 `/signup`
5. 註冊後自動進入產品（網址可能是 `/dashboard`）

**已註冊用戶**
1. 輸入 `daodao.so`
2. 系統偵測已登入
3. 自動顯示產品儀表板
4. 或自動跳轉到 `/dashboard`

#### 優點

✅ **用戶體驗流暢**
- 不會感覺「跳到另一個網站」
- 網址一致，記憶成本低
- 分享連結時就是主域名

✅ **開發維護簡單**
- 只有一個網站
- 設計系統統一
- 減少維護成本

✅ **快速驗證想法**
- 適合新創團隊
- 可以先上線，未來再分離

#### 缺點

❌ **首頁載入較慢**
- 需要判斷登入狀態
- 可能影響 SEO（搜尋引擎看到的是動態內容）

❌ **官網與產品耦合**
- 改官網可能影響產品
- A/B 測試較複雜

❌ **數據分析混雜**
- 訪客和用戶的數據在一起
- 需要用其他方式區分

#### 誰在用這個方案？

**社交平台和社群產品**：
- **Twitter/X** - 未登入看介紹，登入看動態
- **Facebook** - 未登入看官網，登入看動態牆
- **Instagram** - 未登入看精選，登入看完整功能
- **LinkedIn** - 同上
- **GitHub** - 未登入看介紹，登入看儀表板

**為什麼社交平台喜歡這個方案？**
- 內容部分公開（例如：可以看到部分貼文、專案）
- 鼓勵未登入用戶「先看看」，降低註冊門檻
- 產品本身就是內容行銷

### 方案 C：路徑分離 📁 **較少用（10%）**

#### 域名設計

```
https://daodao.so/          → 官網首頁
https://daodao.so/features  → 功能介紹
https://daodao.so/pricing   → 定價
https://daodao.so/app       → 產品入口
https://daodao.so/app/*     → 所有產品頁面
```

#### 優缺點

✅ **優點**：同一個網址，路徑清楚
❌ **缺點**：網址很長（`daodao.so/app/dashboard` vs `app.daodao.so/dashboard`）

#### 誰在用？

較少，主要是一些想要「折衷」的產品。

```

**方案 A（分離）**
```
1. 書籤直接訪問 app.daodao.so
2. 自動登入

體驗評分：⭐⭐⭐⭐⭐ 快速、直接

```

**方案 A（分離）**
```
1. 訪問 daodao.so/pricing
2. 直接看到定價頁面
3. 決定後點擊「開始使用」

| 場景 | 方案 A（分離） | 方案 B（合併） |
|------|---------------|---------------|
| 首次訪問 | ⭐⭐⭐⭐⭐ 清晰專業 | ⭐⭐⭐⭐⭐ 流暢簡單 |
| 老用戶回訪 | ⭐⭐⭐⭐⭐ 直接快速 | ⭐⭐⭐⭐ 需要判斷 |
| 了解產品 | ⭐⭐⭐⭐⭐ 官網內容豐富 | ⭐⭐⭐⭐ 內容可能較少 |
| 分享給朋友 | ⭐⭐⭐⭐ 要說「訪問官網」 | ⭐⭐⭐⭐⭐ 就說「daodao.so」 |
| 記憶成本 | ⭐⭐⭐⭐ 兩個網址 | ⭐⭐⭐⭐⭐ 一個網址 |

**結論**：兩者用戶體驗都很好，差異不大！

## 品牌與行銷考量

### SEO（搜尋引擎優化）

#### 方案 A（分離）- SEO 最佳 ⭐⭐⭐⭐⭐

**優勢**：
```
daodao.so 的內容：
├── 首頁：「線上教育平台」「學習資源分享」 ← Google 容易索引
├── 關於：「我們的故事」「教育理念」        ← 建立品牌
├── 功能：「如何使用島島」                   ← 長尾關鍵字
├── 部落格：「教育趨勢」「學習方法」         ← 內容行銷
└── 合作夥伴：「教育機構合作」               ← 建立信任

結果：Google 搜尋「線上教育平台」時，daodao.so 容易排在前面
```

**為什麼 SEO 好？**
- ✅ 內容完全靜態，載入超快
- ✅ 可以針對每個頁面優化關鍵字
- ✅ 不會有「需要登入才能看」的內容

#### 方案 B（合併）- SEO 較差 ⭐⭐⭐

**劣勢**：
```
daodao.so 的內容：
├── 首頁：動態（未登入看官網，已登入看產品）   ← Google 不確定這是什麼
├── /dashboard：需要登入                      ← Google 看不到
├── /resources：需要登入                      ← Google 看不到
└── /about：可以看到                          ← 只有這些能被索引
```

**為什麼 SEO 較差？**
- ⚠️ 首頁是動態的，Google 不知道該索引什麼
- ⚠️ 大部分內容需要登入，Google 看不到
- ⚠️ 載入速度可能較慢（需要判斷登入狀態）

### 內容行銷策略

#### 方案 A（分離）- 適合大量內容行銷

```
官網 (daodao.so) 可以專注於：
├── 📝 部落格
│   ├── 「如何有效學習」
│   ├── 「教育資源推薦」
│   └── 「教師經驗分享」
│
├── 📖 案例研究
│   ├── 「某某學校使用島島的成功故事」
│   └── 「學生如何透過島島找到資源」
│
├── 🎯 Landing Pages（著陸頁）
│   ├── 針對「老師」的頁面
│   ├── 針對「學生」的頁面
│   └── 針對「家長」的頁面
│
└── 🎉 活動與促銷
    ├── 「開學季活動」
    └── 「合作夥伴招募」

所有這些內容都可以被 Google 索引，帶來自然流量
```

**效果**：每個月可能從 Google 獲得數千到數萬訪客

#### 方案 B（合併）- 較難做內容行銷

```
daodao.so 需要兼顧：
├── 官網內容（給新訪客）
└── 產品功能（給用戶）

部落格和內容可能需要放在：
├── 首頁的一小區塊
└── 或乾脆不放（因為空間有限）
```

### A/B 測試和迭代

#### 方案 A（分離）- 測試容易 ✅

```
想測試不同的首頁標題？
→ 只改官網，產品完全不受影響
→ 可以同時跑多個版本測試

想改變「開始使用」按鈕的顏色？
→ 只改官網，今天改明天就能看數據

想針對不同來源顯示不同內容？
→ 從 Facebook 來的看 A 版本
→ 從 Google 來的看 B 版本
```

#### 方案 B（合併）- 測試複雜 ⚠️

```
想測試不同的首頁標題？
→ 要確保只對「未登入」用戶顯示
→ 不能影響已登入用戶看到的內容
→ 測試邏輯較複雜
```

### 品牌定位

| 方案 | 給人的感覺 | 適合的品牌定位 |
|------|----------|--------------|
| **A（分離）** | 專業、正規、企業級 | B2B SaaS、教育平台、專業工具 |
| **B（合併）** | 親民、輕鬆、社群感 | 社交平台、輕量工具、社群產品 |

## 企業案例學習

### 案例 1：Slack（方案 A）

#### 域名架構
```
slack.com              → 官網
app.slack.com          → 產品
api.slack.com          → API 文件
status.slack.com       → 系統狀態
```

#### 為什麼這樣設計？

**用戶旅程**：
```
1. 聽朋友說 Slack 很好用
2. Google 搜尋「slack」
3. 進入 slack.com
   - 看到標題：「Slack 讓團隊協作更簡單」
   - 看到影片展示
   - 看到客戶案例（Nike、Airbnb 都在用）
4. 被說服，點擊「開始使用」
5. 跳到 app.slack.com 註冊
6. 註冊後創建工作區：yourteam.slack.com
7. 以後都在 yourteam.slack.com 使用
```

#### 學習點

✅ **官網專注於說服**
- 大量的視覺設計
- 客戶案例和社會證明
- 清楚的價值主張

✅ **產品專注於功能**
- 乾淨的介面
- 專注於聊天和協作
- 沒有行銷干擾

✅ **專業形象**
- 大企業看到 slack.com 會覺得「這是個正經的產品」
- 增加信任感

### 案例 2：Twitter/X（方案 B）

#### 域名架構
```
x.com                  → 官網（未登入）+ 產品（已登入）
```

#### 為什麼這樣設計？

**用戶旅程**：
```
1. 朋友分享一則推文連結：x.com/elonmusk/status/123456
2. 點開連結
   - 可以看到這則推文（公開內容）
   - 但功能受限（不能按讚、回覆）
   - 頁面提示「註冊以解鎖更多功能」
3. 好奇心驅使下註冊
4. 註冊後還是在 x.com
   - 但現在可以按讚、回覆、發文
5. 以後打開 x.com 就直接進入動態
```

#### 為什麼社交平台適合方案 B？

✅ **內容即行銷**
- 用戶分享的推文本身就是最好的行銷
- 讓未登入用戶「先看看」，增加註冊慾望

✅ **降低門檻**
- 不需要先看官網介紹
- 直接體驗產品（雖然功能受限）

✅ **病毒式傳播**
- 分享連結時就是主域名
- 簡單好記

#### 學習點

⚠️ **不適合所有產品**
- Twitter 的內容可以公開
- 島島的資源可能需要登入才能看
- 如果內容不能公開，方案 B 的優勢就失效了

- 已登入用戶會被導向自己的工作區

#### 為什麼可行？

✅ Notion 有強大的路由系統
✅ 工作區概念清楚（`/[workspace]`）
✅ 內容可以設定為公開（SEO 仍然有效）

## 常見問題 Q&A

### Q1：如果選方案 A，用戶會不會覺得跳轉很奇怪？

**A**：不會，因為這已經是業界標準。

想想你用過的產品：
- Slack：從 slack.com 跳到 app.slack.com
- Dropbox：從 dropbox.com 跳到產品
- Notion：從 notion.so 跳到工作區

用戶已經習慣了，不會覺得奇怪。

而且，跳轉只發生在「註冊」的時候，之後用戶都直接訪問 app.daodao.so，不會再跳來跳去。

---

### Q2：方案 A 是不是會增加很多工作？

**A**：會增加一些，但長期來看是值得的。

**短期（第一次建立）**：
- 需要設計和開發官網
- 需要撰寫官網內容
- 預估：2-4 週

**長期（日常維護）**：
- 官網內容更新：行銷團隊負責
- 產品功能開發：產品團隊負責
- 兩者互不干擾，反而提高效率

**如果團隊很小（< 5 人）**：
- 可以先用方案 B（快速上線）
- 等團隊長大後再遷移到方案 A
- 技術上不難，主要是規劃

---

### Q3：如果未來想改方案，容易嗎？

**A**：相對容易，但越早決定越好。

**從方案 B 改成方案 A**（較常見）：
```
步驟：
1. 開發新的官網 (daodao.so)
2. 產品遷移到 app.daodao.so
3. 設定重定向：daodao.so/dashboard → app.daodao.so/dashboard
4. 通知用戶（但大部分用戶不會感覺到變化）

時間：2-4 週
風險：低
```

**從方案 A 改成方案 B**（較少見）：
```
步驟：
1. 把產品搬回 daodao.so
2. 官網內容整合進產品
3. 設定重定向：app.daodao.so → daodao.so

時間：1-2 週
風險：中（可能影響 SEO）
```

---

### Q4：用戶會不會記不住網址？

**A**：不會，因為用戶會用書籤或搜尋。

**實際情況**：
- 80% 的用戶會把產品加入書籤（app.daodao.so）
- 15% 的用戶會 Google 搜尋「島島」然後點進去
- 5% 的用戶會直接輸入網址

而且，現代瀏覽器有網址自動完成功能，輸入「dao」就會自動跳出 app.daodao.so。

---

### Q5：SEO 真的那麼重要嗎？

**A**：對教育平台來說，非常重要。

**為什麼？**

教育工作者會這樣找資源：
```
Google 搜尋：
「國小數學教材」
「線上教育平台」
「免費學習資源」
「教學資源分享」
```

如果你們有一個優化良好的官網 + 部落格：
- 這些搜尋都可能找到你們
- 每個月帶來數千到數萬訪客
- 而且是**免費的**自然流量

相比之下：
- Facebook 廣告：需要持續付費
- Google 廣告：每次點擊可能要 $1-5 美金

SEO 是長期投資，但回報很高。

---

### Q6：我們團隊只有 3 個人，應該選哪個？

**A**：短期選方案 B，長期規劃方案 A。

**現在（團隊小）**：
- 用方案 B：daodao.so 同時是官網和產品
- 優點：快速上線，驗證產品
- 目標：找到 Product-Market Fit

**未來（團隊成長到 5-10 人）**：
- 遷移到方案 A
- 增加內容行銷團隊
- 開始認真做 SEO

**時間點**：
- 如果產品已經有 1000+ 活躍用戶
- 或已經拿到投資，準備擴大規模
- 就是遷移的好時機

---

### Q7：部落格一定要獨立域名嗎？可以放在官網嗎？

**A**：可以！有三種選擇：

**選項 1：放在官網**（最簡單）
```
https://daodao.so/blog
```
✅ 優點：簡單，SEO 權重集中在主域名
⚠️ 缺點：如果部落格內容很多，可能讓官網變複雜

**選項 2：獨立子域名**（推薦）
```
https://blog.daodao.so
```
✅ 優點：職責清晰，可以用不同技術（例如 WordPress、Ghost）
✅ 優點：可以讓外部作者貢獻內容

**選項 3：第三方平台**
```
https://medium.com/@daodao
```
✅ 優點：不用維護，專注寫作
⚠️ 缺點：無法完全掌控，SEO 權重不在自己域名

**島島的建議**：
- 初期：用選項 1（daodao.so/blog）
- 未來：遷移到選項 2（blog.daodao.so）
