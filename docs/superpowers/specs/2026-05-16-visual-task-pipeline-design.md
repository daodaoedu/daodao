# Visual Task Pipeline Support

**Date:** 2026-05-16  
**Status:** Approved  
**Scope:** bin/notion-sync/sync.ts, bin/routine-dispatch/verification-loop.sh, .github/workflows/auto-pr-description.yml (monorepo + sub-repos)

---

## Problem

Visual/layout tasks (email templates, UI spacing, RWD) cannot be verified with `pnpm test` вЂ” tests check logic, not pixels. When the pipeline attempts these tasks:

1. Claude makes the correct code change
2. `pnpm test` fails (no visual tests exist)
3. After 2 retries, issue is escalated with `human-coding` label
4. Session time is wasted; the work is discarded

A secondary problem: `auto-pr-description.yml` overwrites pipeline PR bodies with GPT-4o-mini generated content, destroying structured metadata like `Closes #<num>` and review checklists.

---

## Solution Overview

Three coordinated changes:

```
Notion page (contains visual keywords)
  в†“ sync.ts buildLabels()
GitHub issue gets `visual` label
  в†“ pipeline picks up issue
verification-loop.sh detects `visual` в†’ skips pnpm test, runs lint only
  в†“
push branch в†’ open PR (pipeline body preserved)
  в†“ auto-pr-description.yml
detects auto/ branch в†’ skips overwrite
  в†“
Reviewer sees visual review checklist в†’ merges after visual confirmation
```

**File:** `bin/notion-sync/sync.ts`  
**Function:** `buildLabels()`

## Change 2: verification-loop.sh + xs.sh вЂ” Visual Mode

**Files:** `bin/routine-dispatch/verification-loop.sh`, `bin/routine-dispatch/handlers/xs.sh`

`verification-loop.sh` detects `visual` label before the retry loop, sets `VISUAL_MODE=true` env var, then calls the handler. The handler (`xs.sh`) reads `VISUAL_MODE` and switches PR body template accordingly.

**verification-loop.sh вЂ” visual mode behaviour:**
- Query issue labels before calling handler
- If `visual` label present: set `export VISUAL_MODE=true`
- Run handler once (no retry loop)
- Run `pnpm lint` вЂ” still enforced (syntax errors must be caught)
- Skip `pnpm test` вЂ” visual result cannot be machine-verified

## Change 3: auto-pr-description.yml вЂ” Skip Auto PRs

**Files:**
- `.github/workflows/auto-pr-description.yml` (monorepo)
- All sub-repo `.github/workflows/auto-pr-description.yml`

Add an early-exit step before any content generation. Skip the entire workflow if:
- `github.event.pull_request.head.ref` starts with `auto/`, **OR**
- `github.event.pull_request.title` starts with `[auto]`

This preserves the pipeline's structured PR body (including `Closes #<num>`, review checklists, and scope metadata) while leaving the workflow fully active for human-authored PRs.

**Implementation:**
```yaml
- name: Check if auto PR
  id: check_auto
  run: |
    HEAD="${{ github.event.pull_request.head.ref }}"
    TITLE="${{ github.event.pull_request.title }}"
    if [[ "$HEAD" == auto/* ]] || [[ "$TITLE" == \[auto\]* ]]; then
      echo "skip=true" >> "$GITHUB_OUTPUT"
    else
      echo "skip=false" >> "$GITHUB_OUTPUT"
    fi
```

All subsequent steps use `if: steps.check_auto.outputs.skip == 'false'`.

## Out of Scope

- Adding visual/screenshot tests (future work)
- Changing Notion schema (Task Type field can be added later as a second layer)
- Changing scope:S or scope:M handlers (same visual mode logic applies but not in this change)

---

## Risks

| Risk | Mitigation |
|------|------------|
| Keyword false positive (non-visual task gets `visual` label) | lint still runs; worst case is a PR that doesn't need visual review gets the checklist вЂ” harmless |
| Keyword false negative (visual task not detected) | Human can manually add `visual` label to the issue; pipeline re-checks labels at runtime |
| `auto/` branch prefix changes in future | Title check (`[auto]`) is the fallback |
