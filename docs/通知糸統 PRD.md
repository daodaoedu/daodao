## **Overview**

通知系統的核心任務並非僅是發送訊息，而是作為行為設計的觸發器，引導用戶回到平台參與討論、確認進度，並獲得來自社群的認同感（正向回饋循環）。

## **Objectives & Goals**

* **提升回流率 (Retention)：** 透過即時且具溫度的社交觸發器，引導用戶回到平台參與對話。  
* **管理認知負荷：** 透過「聚合邏輯」與「細粒度設定」，防止資訊過載造成的數位疲勞。  
* **強化敘事一致性：** 統一通知文案的口吻，將「收藏、關注、連結」等動作融入島嶼探索的語境中。  
* **縮短互動閉環：** 確保從接收通知到進行回覆（反應或留言）的摩擦力降至最低。

## **Personas & User Stories**

| 角色 | 用戶故事 (User Story) |
| ----- | ----- |
| **積極學習者**  | 作為一名重視討論的專業人士，我希望在有人回覆我的意見或發送「連結」請求時收到即時提醒，以免錯過建立高品質夥伴關係的時機。 |
| **目標導向者**  | 作為一名時間碎片化的學習者，我希望在非工作時段關閉瑣碎的反應通知，但每週收到一封「學習成果摘要」Email，讓我知道我的內容對他人有幫助。 |
| **追求品質者**  | 作為一名重視隱私的學習者，我希望可以設定誰能對我發送「連結」通知，確保我的通知中心只保留對我有價值的連結信號。 |

## **Features and Functionalities**

### **通知觸發類型**

| 互動類型 | 預設頻道 |
| ----- | ----- |
| **反應 (Reaction)** | In-App |
| **留言 (Comment)** | In-App \+ Email |
| **關注 (Follow)** | In-App |
| **連結請求 (Connect)** | In-App \+ Email |
| **連結確認** | In-App |

### **通知管道** 

* **App 內通知 (In-App)：** 鈴鐺圖示，顯示通知列表；小鈴噹，打開信箱，有分已讀未讀  
* **電子郵件 (Email)：** 針對高價值的互動（如：新留言、新追蹤）發送摘要或即時郵件

### **通知設定**

* **全局開關：** 一鍵開啟/關閉所有通知。  
* **分項控制：** 用戶可針對「讚/檢舉/留言/追蹤/收藏」分別設定 In-App 與 Email 的發送頻率。

## **UX Notes**

* **情感化文案：** 避免使用「系統通知」等冰冷字眼。  
* **深層跳轉 (Deep Linking)：** 點擊留言通知應準確滾動至該留言位置並高亮 (Highlight) 顯示 2 秒。  
* **連結請求透明化：** 在連結請求通知中，直接顯示對方的「連結初衷」摘要，減少用戶點擊進去才能查看的負擔。  
* **無縫操作：** 連結請求通知應包含「接受」與「忽略」的快速操作按鈕。

## **Release Criteria**

* **連結成功率：** 連結請求的通知送達成功率需達到 **99.9%**，且 Email 發送延遲不超過 1 分鐘。  
* **設定一致性：** 當用戶在 App 內修改通知設定後，必須即時反應在該用戶的通知推送佇列中，不得發送已被關閉類別的 Email。  
* **退訂功能：** 所有的 Email 通知必須包含清晰的「取消訂閱」或「通知設定」連結。
